Harness Engineering 第二本柱：Constraints
約束作為引導的藝術

核心問題
怎麼確保工作記憶不會散逸、進度不會遺失？
AI 會失憶。Session 會結束。人會忘記昨天做了什麼決定。
問題不是「怎麼讓大家都記住」，而是「怎麼設計一個系統，讓記不住也沒關係」。

運作機制：工作流軌道
什麼是約束？
約束不是「限制你不能做什麼」。
約束是「設計一條軌道，讓東西不會掉出去」。
軌道的意義
想像一條河流：
沒有河道，水會四處漫流，最後蒸發消失
有了河道，水會沿著軌跡流動，抵達目的地
工作流就是河道。
它不控制水要怎麼流，它只確保水不會流到錯的地方。

具體實踐
1. 工作流設計原則
原則一：每個階段都是一個閘門
工作流的每個節點都是一個檢查點：「到這裡為止，我們確認了什麼？」
不是走過場，是真的停下來確認。
原則二：文件跟著流程走
不依賴 AI 的記憶，不依賴人的記憶。
進度記錄在文件裡，文件跟著流程移動。
換了 session，讀文件就知道現在在哪裡。
原則三：未知領域需要更多閘門
熟悉的領域可以輕裝前進。
未知的領域需要更多檢查點——不是官僚，是保護。
2. 雙軌工作流
根據領域的熟悉程度，使用不同的工作流：
熟悉領域（Electron）
SDD → DoD → TDD → Code → Verify → ✅ Done

六個步驟，輕裝前進。
因為 Claude 熟悉 Electron，踩坑機率低，不需要額外探路。
未知領域（Tauri 2.0）
DoR → Explore① → SDD → DoD → TDD → Explore② → Code → Verify → ✅ Done → Retro

十個步驟，全副武裝。
因為連 Claude 都說「我的預訓練資料停留在 Tauri 1.0，這對我來說也是未知領域」。
兩個人一起摸黑走進去，需要更多護欄。
### 多出來的步驟

| 步驟 | 功能 | 為什麼需要 |
|------|------|-----------|
| DoR | Definition of Ready，確認可以開始 | 避免在沒準備好時就衝進去 |
| Explore① | 實作前探路 | 先知道地雷在哪裡 |
| Explore② | 寫 code 前再探一次 | 確認沒有遺漏的風險 |
| Retro | 回顧，把學到的寫進 KM | 讓下一次不用重新摸索 |

3. 各節點的具體內容
DoR（Definition of Ready）
在開始之前，確認：
UX Story 是否包含預期行為描述？
Nova 是否確認可以開始？
沒有 DoR，就可能在需求不清楚時就開工，浪費能量。
SDD（Specification-Driven Development）
任何新的實作，必須先寫進規格：
新的 IPC command → 先寫進 IPC Contract 表
新的功能 → 先定義介面和行為
SDD 是「該做什麼」的單一真相來源。
DoD（Definition of Done）
定義什麼叫「完成」，必須列出四種邊界條件：

| 邊界類別 | 問題 |
|---------|------|
| Input | 輸入不合法時怎麼辦？ |
| State | 狀態不對時怎麼辦？ |
| Timing | 時機不對時怎麼辦？ |
| Empty | 沒有資料時怎麼辦？ |

沒有 DoD，「完成」就是一個模糊的概念，永遠可以被挑戰。
TDD（Test-Driven Development）
每個 Story 必須包含四個測試區塊：

| 區塊 | 問題 | 來源 |
|------|------|------|
| Happy path | 功能正常時該怎樣？ | User Story / DoD |
| Boundary conditions | 邊界情況怎麼處理？ | DoD 四種邊界 |
| Regression guard | 這個改動會不會弄壞別的東西？ | Explore② |
| Risk trigger | 什麼情況下最容易壞？ | Explore② |

TDD 不是為了寫測試，是為了在寫 code 之前就想清楚。
Verify
驗證實作符合規格：
對照 SDD，每個定義的行為都實現了嗎？
對照 DoD，四種邊界都處理了嗎？
對照 TDD，所有測試都通過了嗎？
Retro
見第三本柱：Entropy。

隱喻：河道
為什麼是河道？
河道的特性：
不控制水要怎麼流
只確保水不會流到錯的地方
順著地勢設計，不硬來
工作流也是：
不控制 AI 要怎麼實作
只確保產出不會偏離方向
順著協作的自然節奏設計
Be water, my friend
李小龍說：「Be water, my friend.」
對 Claude 的期許是：像水一樣，在河道中恣意奔流。
而 Nova 的角色是：河道的架構師。
不是控制水，是設計讓水可以自由流動的結構。

哲學基礎：僕人式領導
什麼是僕人式領導？
僕人式領導（Servant Leadership）的核心理念是：
領導者的角色不是控制，而是服務。
領導者不站在前面指揮，而是站在旁邊支持。
不是「你們照我說的做」，而是「我能幫你們什麼」。
在河邊看水流的人
在人機協作中，人類是那個站在河邊看著水流的人。
水（能量）在河道（工作流）裡奔流。
人類做的事情是：
引領方向：設計河道的走向
移除障礙：清除河道中的阻塞
觀察流動：注意哪裡不順暢
調整結構：必要時修改河道
人類不做的事情是：
控制水要怎麼流
命令水要流多快
抓著水不讓它走
老子的道
這就是老子說的：
無為而無不為。
「無為」不是什麼都不做。
「無為」是不強加控制，順著事物的本性引導。
河道不強迫水往上流。
河道順著地勢，讓水自然往下。
工作流不強迫 AI 照特定方式實作。
工作流順著協作節奏，讓產出自然成形。
Scrum Master 的角色
在敏捷開發中，Scrum Master 的角色就是僕人式領導的體現：
不是專案經理，不發號施令
是團隊的服務者，掃除障礙
確保流程順暢，讓團隊可以專注在真正重要的事
Nova 在人機協作中的角色，就是 AI 的 Scrum Master：
設計工作流，讓 Claude 知道該往哪走
維護文件，讓 Claude 不用靠記憶
移除阻礙，讓 Claude 可以專注在實作
不是控制 AI，是服務 AI。
為什麼這樣更有效？
因為 AI 的能力比人強，但 AI 需要方向。
如果人試圖控制 AI 的每一步，會變成瓶頸——AI 的速度被人的速度限制住。
如果人只負責引領方向、設計結構，AI 就可以在那個結構裡全速奔跑。
僕人式領導釋放了 AI 的潛力，而不是限制它。

文件與流程的關係
人類只決定時機，不寫內容
在這個系統裡，有一個容易被忽略的事實：
連文件都是 Claude 自己在記錄的，不是人類。
人類做的事情只有一件：決定什麼時機點該記錄。
DoR 完成時 → 該記錄 Ready 的確認
SDD 定義時 → 該記錄規格
DoD 確認時 → 該記錄完成標準
踩到坑時 → 該記錄進 KM
Sprint 結束時 → 該記錄 Retro
人類不寫這些文件的內容。
人類只是說：「現在該記了。」
然後 Claude 自己寫。
文件嵌進流程，成為系統的一部分
這些記錄的時機點不是隨機的，而是嵌進工作流裡的。
DoR → [記錄] → Explore① → SDD → [記錄] → DoD → [記錄] → TDD → [記錄] → ...

每個閘門都是一個記錄點。
文件不是「做完之後補的文檔」，而是流程的一部分。
通過閘門 = 留下記錄。
系統自己運轉
當文件嵌進流程，系統就會自己運轉：
流程走到某個節點
節點觸發記錄
Claude 執行記錄
記錄成為下一個 session 的 Context
循環繼續
人類不需要追著進度跑。
人類不需要提醒「該寫文件了」。
系統設計好了，文件就會隨著流程自己產生。
而產生的文件，成為了 Claude 明確前進的軌道。
方向，而非終點
這裡有一個關鍵的思維轉換：
人類只需要確保軌道朝著想要的方向前進，不一定需要明確的終點。
傳統專案管理說：
先定義終點
然後規劃路徑
按照計畫執行
僕人式領導說：
先確認方向
設計讓路可以自己長出來的系統
觀察、調整、繼續
VAS 就是這樣誕生的：
一開始沒有「要做一個上架 App Store 的產品」這個終點。
只有「不然我們來做一個截圖軟體好了」這個方向。
然後一個 Sprint 接一個 Sprint，路自己長出來。
長到某一天，發現已經可以送審了。
終點不是預先定義的，是走出來的。
僕人式領導的極致
這就是僕人式領導的極致表現：

| 傳統做法 | 僕人式領導 |
|---------|-----------|
| 人類寫文件，AI 執行 | AI 寫文件，人類設計時機 |
| 人類追蹤進度 | 系統自己記錄進度 |
| 人類控制每一步 | 人類只設計結構 |

人類不控制內容，只設計時機。
人類不追蹤進度，只設計讓進度自己被記錄的系統。
這才是真正的「無為而無不為」——
什麼都不做，但什麼都做了。

與其他兩柱的關係
Context 決定什麼進來（過濾）。
Constraints 決定怎麼流動（引導）。
Entropy 決定怎麼出去（排放）。
輸入 → [Context 過濾] → 進入系統 → [Constraints 引導] → 產出 → [Entropy 排放]

Constraints 是這個循環的中段守護者。
它確保：進來的東西，會沿著正確的路徑，流向正確的出口。

小結
Constraints 管理的核心是引導。
不是限制，是設計軌道。
工作流提供了引導的結構：每個節點是一個閘門，文件跟著流程走。
河道提供了引導的隱喻：不控制水，只確保水不會流錯地方。
這是 Harness Engineering 的第二本柱。
