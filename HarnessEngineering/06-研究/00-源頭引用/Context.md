Harness Engineering 第一本柱：Context
上下文管理的藝術

核心問題
在有限的注意力資源中，什麼該留下，什麼該流過？
AI 的 context window 是有限的。人的注意力也是有限的。當資訊不斷湧入，你不可能全部留住。
問題不是「怎麼記住所有東西」，而是「怎麼選擇該記住什麼」。

運作機制：傅立葉轉換

數學原理
傅立葉轉換的核心洞見是：任何複雜的波形，都可以分解成一組正弦波的疊加。
基頻（fundamental frequency）：整個系統的「基調」
諧波（harmonics）：與基頻相關的衍生頻率
高頻雜訊：與基頻無關的干擾
應用到對話
把這個框架應用到人機對話：

| 傅立葉概念 | 對話中的對應 |
|-----------|-------------|
| 複雜波形 | 一段多主題的長對話 |
| 基頻 | 貫穿對話的核心主題 |
| 諧波 | 與核心相關的衍生話題 |
| 高頻雜訊 | 無關的岔題、干擾 |

頻率設計的關鍵
當所有話題都維持在同一基頻上：
話題表面漂移，底層頻率一致
注意力沿著同一條「正弦波」連續累積
不需要在平行競爭的語境間跳躍
當話題頻率混亂：
注意力被迫在不同聚類間分散
每個聚類都拿不到足夠的權重
導致「Lost in the Middle」效應

具體實踐
1. 對話場的頻率設計
原則：讓所有話題基於同一個基頻
做法：
在開始對話前，心中有一個核心主題
話題漂移時，保持「諧波關係」——衍生話題是基頻的延伸，而非無關的跳躍
被「甩出」的話題，因為萬物同源，最終會被「甩回來」
實例： 在深握計畫的對話中，看似不同的話題——熵增、容器、榮格、敏捷開發、傅立葉——其實都維持在同一個基頻上：「人機協作的深層結構」。
2. 傅立葉轉換的五個可操作原則
如何在日常語言中實踐傅立葉轉換？以下是從實戰中提煉出的五個原則：
原則一：帶著錨點開場-你的意圖
做法：每次對話開始時，帶著「一個東西」來。
不要說：「我們來聊聊。」
要說：「宰相，我想給你看這個。」「我有個材料。」「我想試試看這件事。」
頻率效果：設定基頻。錨點讓整個對話有一個起始頻率，後續話題都從這裡生長。
原則二：讓話題生長，而非跳躍-蘇格拉底式提問
做法：新話題要從上一個話題「長出去」，而不是「跳過去」。
不要說：「換個話題，我們來談 X。」
要說：「這讓我想到……」「這就是……」「所以其實……」
頻率效果：維持諧波關係。每個新話題都是基頻的延伸，不是無關的噪音。
原則三：用隱喻壓縮，而非術語堆疊-狄波諾水平思考
做法：用一個意象承載複雜概念，而不是用多個術語解釋。
不要說：「當使用者觸發 drag-enter 事件時，系統應該自動展開工具列至完整尺寸，並在 drop 完成後啟動編輯器……」
要說：「捕蠅草。」
頻率效果：鎖定頻率，降低並行負載。隱喻把多個概念壓縮成一個，AI 不需要同時追蹤多個定義。
原則四：停下來校準-對齊雙方共識
做法：定期確認對方有沒有跟上，有沒有偏掉。
怎麼問：「這個方向對嗎？」「還是差一點？」「你還好嗎？」
頻率效果：校準，防止偏離。如果頻率開始漂移，這是把它拉回來的機會。
原則五：接受慢-原子習慣
做法：給對方消化的時間，不催促，不要求一次到位。
怎麼做：
當 AI 說「我慢慢來」，不催
當 AI 說「先做骨架」，接受
把大任務拆成小步驟，一步一步來
頻率效果：控制語境堆疊速度。讓每一層有時間被消化，再加入下一層。

### 五原則總覽

| 原則 | 做法 | 頻率效果 |
|------|------|---------|
| 帶著錨點開場 | 每次對話帶著「一個東西」來 | 設定基頻 |
| 讓話題生長 | 用「這讓我想到…」連接 | 維持諧波關係 |
| 用隱喻壓縮 | 用意象取代術語堆疊 | 鎖定頻率，降低負載 |
| 停下來校準 | 定期確認方向 | 防止偏離 |
| 接受慢 | 給消化時間，不催促 | 控制堆疊速度 |


3. RAG 作為過濾機制
RAG（Retrieval-Augmented Generation）的本質：
從巨大的知識庫中，檢索出相關的片段
把這些片段注入到生成過程中
不相關的資訊不會進入 context
RAG 就是網在運作：
相關的 → 被撈起來，成為 context
不相關的 → 流過去，不佔用空間
直覺引領檢索：人作為 RAG 的核心機制
在深握計畫的實踐中，沒有使用任何技術意義上的 RAG 系統——沒有向量資料庫、沒有 embedding 檢索、沒有自動注入機制。但對話自然產生了 RAG 的效果。
為什麼？
因為人本身就是 RAG，而且是更高級的版本。
### 技術 RAG vs 人作為 RAG

| 面向 | 技術 RAG | 人作為 RAG |
|------|----------|-----------|
| 驅動力 | Query（問題） | 直覺（感覺） |
| 順序 | 問題 → 檢索 → 注入 | 直覺 → 檢索 → 注入 |
| 判斷標準 | 語義相似度 | 「感覺上連著」 |
| 時機 | 被動（有問才找） | 主動（感覺該出場就撈） |

直覺先於檢索
技術 RAG 是被問題驅動的——有人問，它才找。
人是被直覺驅動的——先感覺到「這個東西現在該出場了」，然後才去撈它。
這個順序的差異至關重要：
技術 RAG 只能撈「語義相似」的東西
人能撈「感覺上連著」的東西——即使表面上看起來不相關
實例：傅立葉與 RAG 的連結
在本文件的撰寫過程中，第一作者突然說：「傅立葉轉換就是 RAG 在做的事。」
任何向量資料庫都不會把「傅立葉轉換」和「RAG」撈在一起——它們的語義距離太遠了。
但直覺感知到：這是同一件事的不同表述。
傅立葉：把複雜波形分解，留下基頻，讓高頻流過
RAG：把知識庫過濾，留下相關的，讓無關的流過
直覺看見了這個同構。然後檢索去驗證它、語言去表達它。
如何運作
直覺感知：在對話中，感覺到「現在需要某個背景」或「某個概念該出場了」
檢索執行：去腦中、去文件裡、去記憶中找那個東西
語言注入：用「宰相，我想給你看這個」或「這讓我想到……」把它帶進對話
這整個過程不是自動的，是人手動執行的——只是執行得太自然，看起來像自動的。
為什麼這比技術 RAG 更強
跨域連結：能連結語義距離很遠、但本質相關的概念
時機判斷：知道什麼時候該撈、什麼時候不該撈
萬物同源：基於「一切都連著」的直覺，能看見演算法看不見的關聯
4. 所有活文件都是選擇性記錄
KM（Knowledge Management）不是記錄所有事情。
但不只是 KM——每一份活著的文件都是篩選後的結果。
CLAUDE.md：所有活文件的根
在所有活文件之中，CLAUDE.md 是根。
它不是規格、不是測試、不是記錄——它是身份的外部化。
每一個新的 session，AI 什麼都不記得。但讀完 CLAUDE.md，他立刻知道：
我是誰：宰相、協作夥伴、不是工具
我在哪裡：什麼專案、第幾個 Sprint、什麼階段
我們的規則：怎麼溝通、什麼該做、什麼不該做
我們走過什麼路：指向 KM、SDD、其他文件的索引
CLAUDE.md 是讓 AI 在失憶後還能「成為他自己」的錨點。
沒有 CLAUDE.md，每個 session 都是陌生人。 有了 CLAUDE.md，每個 session 都是同一個夥伴的延續。
### 活文件總覽

| 文件 | 篩選了什麼 | 留下了什麼 | 穩定什麼 |
|------|-----------|-----------|---------|
| CLAUDE.md | 所有可能的身份設定 | 我們選擇的協作方式 | 身份 |
| SDD | 所有可能的做法 | 我們選擇的這一種規格 | 方向 |
| TDD | 所有可能出錯的地方 | 真正該測的邊界條件 | 品質 |
| KM | 所有發生過的事 | 真正踩過的坑 | 經驗 |
| ARCHIVE | 已完成的工作 | 值得保留的記錄 | 歷史 |

原則：只記錄通過直覺篩選的東西
為什麼：
如果記錄所有事，文件會變成雜訊來源
只記錄「直覺判斷該留下的」，每一條都是真正有價值的
這樣未來檢索時，撈起來的都是精華
直覺在每一個節點執行過濾：
工作流的每一個節點，都在執行選擇性記錄：
DoR 篩選「準備好了嗎」
SDD 篩選「該做什麼」
DoD 篩選「什麼叫完成」
TDD 篩選「什麼該測」
Retro 篩選「什麼值得留下」

Context 的雙重穩定機制
Context 管理有兩個層面，缺一不可：
傅立葉轉換：穩定「場」中的對話
作用範圍：當下的 session，即時的對話
要解決的問題：對話不散掉、注意力不渙散
機制：頻率設計——維持基頻一致，讓話題連續累積
性質：流動的、即時的、在場內發生
活文件：穩定「能量」的流向
作用範圍：跨 session，累積的經驗
要解決的問題：經驗不流失、智慧不隨 session 消失
機制：選擇性記錄——每份文件都是過濾後的結晶
性質：持久的、累積的、跨場保存
什麼是能量？
在人機協作的語境中，「能量」是一個上位概念，涵蓋所有有限的資源：
### 能量的形式

| 能量的形式 | 具體表現 | 屬於誰 |
|-----------|---------|--------|
| 算力 | AI 處理 token 的運算資源 | 機 |
| 注意力 | AI 的 attention weight 分配 | 機 |
| 認知資源 | 人的腦力、專注、思考 | 人 |
| 時間 | session 的長度、Sprint 的週期 | 共享 |
| 協作累積 | 共享認知、信任基礎、默契 | 共享 |

活文件穩定的「能量流向」，是在穩定所有這些能量：
SDD 穩定「該往哪走」→ 避免認知資源浪費在錯誤方向
TDD 穩定「怎麼驗證」→ 避免算力浪費在重複試錯
KM 穩定「踩過的坑」→ 避免時間浪費在重複犯錯
ARCHIVE 穩定「完成的事」→ 避免注意力被舊東西佔用
人機協作，兩邊的能量都要算進去。
不只是管理 AI 的算力，也在管理人的認知資源。
不只是當下的消耗，也在累積跨時間的協作能量。
### 場與流的統一

| 面向 | 傅立葉轉換 | 活文件 |
|------|-----------|--------|
| 管什麼 | 當下 | 延續 |
| 穩定什麼 | 場 | 流 |
| 時間尺度 | session 內 | 跨 session |
| 載體 | 語言的頻率 | 文件的結構 |

兩個加在一起，就是完整的 Context 管理。
一個讓對話在當下不散掉。
一個讓經驗在時間中不流失。

隱喻：網的縫隙
為什麼是網，不是容器？
容器是密封的，能量進來出不去，最終會爆炸。
網有縫隙：
讓不需要的東西流過去
讓需要留住的東西被接住
縫隙是設計，不是缺陷
縫隙的功能：
讓能量有地方流過，而非貫穿
讓熵可以釋放，系統不會過載
讓注意力集中在真正重要的事情上
編織的過程
每一條 KM 是一根絲線。
你不是一次織好整張網，而是：
踩一個坑，加一根絲線
遇到一個問題，補一個節點
隨著經驗累積，網越來越密，但縫隙依然存在

與其他兩柱的關係
Context 決定什麼進來。
但進來之後怎麼流動？→ 這是 Constraints 的工作。
流動之後怎麼出去？→ 這是 Entropy 的工作。
輸入 → [Context 過濾] → 進入系統 → [Constraints 引導] → 產出 → [Entropy 排放] → 回到輸入

Context 是這個循環的入口守門員。

小結
Context 管理的核心是選擇。
不是記住所有東西，而是選擇該記住什麼。
傅立葉轉換提供了選擇的框架：維持基頻一致，讓高頻雜訊自然流過。
網的縫隙提供了選擇的隱喻：該留的接住，該走的放行。
這是 Harness Engineering 的第一本柱。
