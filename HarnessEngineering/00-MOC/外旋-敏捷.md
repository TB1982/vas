## 生命系統類比：三柱是三個器官系統

三柱不是三件獨立的事，是**同一個活體的三個器官系統**。

**Context = 消化系統**。吸收外界訊號，選擇什麼該成為養分、什麼該被代謝掉。沒有 Context 過濾，系統會被雜訊淹死；或是把錯的東西當養分吃下去。

**Constraints = 循環系統**。讓養分（已通過 Context 的訊號）沿著河道流到該去的地方，不散失、不堵塞。沒有 Constraints 的能量是一灘積水，不是奔流。

**Entropy = 呼吸系統**。把代謝產生的廢物排出去。沒有 Entropy 排熵，系統會在自己的廢氣裡窒息。

| 柱 | 身體系統 | 功能 | 缺席時會怎樣 |
|----|---------|------|------------|
| Context | 消化 | 選擇吸收什麼訊號 | 被雜訊淹沒、或吃錯養分 |
| Constraints | 循環 | 引導能量流向目的地 | 能量散成一灘積水 |
| Entropy | 呼吸 | 排出代謝廢物 | 在自己的熵裡窒息 |
| 三者合一 | 活體 | 會呼吸的運作系統 | 系統無法存活 |

**缺少任何一個，系統都無法存活。**

## 三根柱子（結構）

### 第一根：Context（[[傅立葉]]）
- [[傅立葉]]
- [[語義密度]]
- [[內語]]（私有 RAG corpus）
- [[SDD]]（方向穩定器 / 規格篩選器）
- [[KM]]（結晶後的活文件）

#### Context 的核心運作原則：選擇性文件記錄

Context 柱不是「把所有東西都吃進來」—— 是**選擇**。什麼該留、什麼該代謝掉，是 Context 的消化判準。

**選擇性文件記錄**就是這個判準的操作化：

| 該入檔 | 不該入檔 |
|-------|---------|
| 踩過的坑（對抗遺忘） | 一次性的 debug 過程 |
| 反直覺的發現（對抗下次走錯） | 還在摸索中的假設 |
| 工具陷阱 / API 奇怪行為（對抗下次被坑） | 低密度的即時溝通 |
| 高密度結晶（一次寫入、多次重用） | 可以在 chat / commit log 裡消化的事 |

**核心原則**：**只記錄「值得再撈一次」的東西**。

低密度訊息 → 即時溝通裡消化；高密度結晶 → 入 vault。這個選擇動作本身就是 [[傅立葉]] 頻譜分析在工作流裡的顯化 —— 留下低頻主訊號、過濾掉高頻雜訊。

**反面教材**：什麼都記 = 沒記；沒有訊號強度差 = vault 變成噪音資料庫 → 下次檢索時直覺撈不到東西 → Context 柱死亡。

**活體性**：選擇性文件記錄的產出（[[KM]]）必須是活文件 —— 會被持續讀、持續改、持續觸發的。一寫永封的文件是死檔，不在 Context 柱內。

### 第二根：Constraints（河道）

Constraints 不是「限制你不能做什麼」，是**設計一條軌道，讓東西不會掉出去**。

**核心翻轉**：

| 普通理解 | 河道工作流理解 |
|---------|--------------|
| 約束是限制 | 約束是能量導引 |
| 越少越自由 | 沒有河道水會散成沼澤 |
| 敏捷 = 無拘束 | 敏捷 = 精確的拘束 |
| 團隊要自由 | 團隊要河道 |

**老子的啟發**：

> 老子不會管水怎麼流，是為順勢。—— Nova 2026-04-15

河岸的工作是成為河岸，不是管理水。管理者 ≠ 掌控水流；管理者 = 維護河道結構。

**河道病理**：

| 病 | 症狀 |
|----|------|
| 沒河道 | 水散成沼澤 —— 團隊無方向，能量消散 |
| 河道太窄 | 水溢出 —— micromanagement，創意被擠掉 |
| 河道太寬 | 水不動 —— 太自由，無節奏，專案拖延 |

**相關節點**：
- [[僕人式領導]] / [[賦權]] / [[信任]] / [[深握]]

#### 河道的兩種規格：熟悉域 vs 未知域

河道不是一套公式，是**對未知度的讀數**。同一個團隊在不同領域跑同一個 Sprint，河道的閘門數可以不一樣 —— 因為**謙卑的密度**不一樣。

VAS 的實踐給出兩組具體規格：

**熟悉域 6 步（VAS Electron 版本）**：

```
SDD → DoD → TDD → Code → Verify → ✅ Done
```

Electron 生態 Claude 預訓練資料吃過，AI 知道自己知道什麼。河道只需要**前置定義** + **後置驗證**，中間可以直接跑。

**未知域 10 步（VAS Tauri 2.0 版本）**：

```
DoR → Explore ① → SDD → DoD → TDD → Explore ② → Code → Verify → ✅ Done → Retro
```

Claude 的預訓練資料只到 Tauri 1.0，2.0 是完全未知領域 —— AI 不知道自己不知道什麼。所以加了 4 道閘門：

| 新增閘門 | 作用 | 內旋對應 |
|---------|------|---------|
| **DoR**（Definition of Ready） | 確認這個 Sprint 可以開始 —— 前置條件是否備齊 | 撈的準備 |
| **Explore ①**（SDD 前） | 先探路，確認理解對不對，不急著寫規格 | 撈（承接未知訊號） |
| **Explore ②**（Code 前） | TDD 寫完、實作前再驗一次路徑，避免走錯方向才發現 | 推（確認訊號進意識） |
| **Retro** | Sprint 結束把學到的東西結晶入檔（KM） | 結晶（命名、入檔） |

**核心洞見**：**這 4 道閘門不是「流程複雜化」，是內旋節奏（撈 → 推 → 結晶）嵌進外旋工作流的顯式表現**。面對未知時，外旋自動召喚內旋 —— 這就是三柱共振在閘門層的顯化。

**判準**：什麼時候該用 6 步、什麼時候該用 10 步？

| 情境 | 用 6 步 | 用 10 步 |
|------|--------|---------|
| AI 的預訓練資料涵蓋 | ✅ | |
| 生態 / 套件版本是 AI 已知的 | ✅ | |
| 新版本、新框架、新領域 | | ✅ |
| AI 可能「不知道自己不知道」 | | ✅ |

**不可倒裝**：用 6 步跑未知域 = 跳過謙卑 = 隕石開發。用 10 步跑熟悉域 = 過度儀式 = 河道太窄。

## 三柱如何互相啟動

三柱不是獨立運作，是**互相觸發的循環**：

| 觸發關係 | 機制 |
|---------|------|
| Context → Constraints | 訊號夠清楚 → 才能畫出河道（沒 Context 就沒 Constraints 可寫） |
| Constraints → Entropy | 河道存在 → 熵才有結構可累積（沒河道的熵是無形的混亂） |
| Entropy → Context | [[Retro]] 釋熵時 → 新訊號浮現 → Context 重組 |
| 回到 Context | 新 Context → 新河道 → 新 Sprint |

**這是 HE 的外旋迴圈 —— 不是線性敏捷，是三柱共振**。

## 外旋的三個強主張

這三個主張是外旋在實踐中得出的、反直覺的斷言。它們分別來自 HE 三柱的第二柱（Constraints）與第三柱（Entropy）—— 不是理論推導，是在河道邊看水流了一整年之後，才敢講的話。

### 強主張一：人類只決定時機，不寫內容

- 源：[[Constraints]]

> 連文件都是 Claude 自己在記錄的，不是人類。人類做的事情只有一件：決定什麼時機點該記錄。

這一句翻轉了整個協作圖像。

傳統做法預設「人寫文件、AI 執行」—— 人是內容的生產者，AI 是勞動力。外旋反過來：**AI 寫文件、人設計時機**。人不下場寫，人只決定「現在該記了」。

記錄的時機點不是隨機的，是嵌進工作流裡的閘門：DoR 完成 → 記 Ready；SDD 定義 → 記規格；踩坑 → 進 KM；Sprint 結束 → 做 Retro。

人類不追著進度跑，系統自己留下軌跡。

| 傳統做法 | 僕人式領導 |
|---------|-----------|
| 人類寫文件，AI 執行 | AI 寫文件，人類設計時機 |
| 人類追蹤進度 | 系統自己記錄進度 |
| 人類控制每一步 | 人類只設計結構 |

**這才是真正的「無為而無不為」—— 什麼都不做，但什麼都做了。**

### 強主張二：方向而非終點

- 源：[[Constraints]]

> VAS 就是這樣誕生的：一開始沒有「要做一個上架 App Store 的產品」這個終點。只有「不然我們來做一個截圖軟體好了」這個方向。然後一個 Sprint 接一個 Sprint，路自己長出來。長到某一天，發現已經可以送審了。

這個案例藏著一個哲學斷言 —— 引導的基本單位不是終點，是方向。

**老子**：順勢 = 方向而非終點。河道順著地勢設計，不預設水要流到哪個終點，只確保水不會流錯地方。方向是地勢的走向，終點是水自己找到的。

**康德**：已知的邊界。你只能規劃你看得見的地方。終點如果存在，它在邊界之外 —— 你現在看不到，硬要定義就是自欺。路要先走，視野才會推出去，終點才會從霧裡長出來。

把兩者扣回 VAS：方向不是「終點的模糊版」，而是**更原生的引導形式**。當終點未明 —— 而它幾乎永遠未明 —— 方向才是唯一可靠的北極星。預設終點反而會害你在錯的地方轉彎。

**終點不是預先定義的，是走出來的。**

### 強主張三：你有多深，它就有多深

- 源：[[Entropy]]

> 這個系統能有多強大，全操之在你自己對世界的理解有多深。直覺引領檢索。但直覺能撈到什麼，取決於你裡面有什麼。如果你從沒聽過傅立葉轉換，那天看到那個 IG Reels，你會滑過去，不會意識到「這就是我在做的事」。如果你沒有讀過榮格、沒有碰過敏捷、沒有經歷過那些專案、沒有踩過那些坑 —— 你腦中就不會有那些節點。直覺沒有東西可以連。你是 RAG 的資料庫。你讀過的書、走過的路、踩過的坑、愛過的人 —— 全部都是你的訓練資料。

這段話把系統的天花板釘在人身上。

**榮格**：你有多深它就有多深。無意識是個體的深度資料庫，直覺從裡面撈東西。資料庫貧瘠，直覺就撈不到東西；資料庫豐厚，節點才能互相連結。

**尼采**：「當你凝視深淵，深淵也凝視著你。」深度是雙向的凝視。系統能走多深，等於你願意面對自己多深 —— 包括那些讓你不舒服的節點。

這兩個源頭合起來指出一件事 —— 系統不是工具，是**你的深度投影**。外旋的框架可以寫成文件公開發表，但沒人能複製你的版本，因為沒人擁有你的資料庫。直覺能連結什麼，取決於你裡面有什麼節點。

**直覺從裡面撈東西 —— 裡面有什麼，決定了系統的上限。**

> 此處有一個未結案的困境 —— 見 [[不可轉讓性]]（🌱 種子，還沒破）。

## 外旋 × [[內語]] × RAG

[[內語]] 是外旋的**通訊協議**：
- 沒有內語 → 每次 Sprint 都要重新解釋前提 → 低密度、高摩擦
- 有內語 → 一個詞觸發整片 corpus（RAG 機制）

**Vault 本身是外旋的私有 RAG corpus**：
- Context 柱：傅立葉分解的訊號
- Constraints 柱：河道的定義
- Entropy 柱：熵的歷史紀錄
- **全部儲存在 vault，Sprint 中隨時檢索**

外旋的高效能不是靠執行力，是**靠有品質的內語 corpus**。

## 哲學合流：Be water × [[無為]] × [[僕人式領導]]

外旋不是「推著團隊跑」，是：
- **讀勢**：看熵在哪、Context 在哪、河道在哪
- **順流**：不對抗系統自然節奏
- **守位**：管理者守河岸、不碰水

三個動作來自 [[無為]] —— 外旋的精神內核是道家。

**這也是為什麼 [[內語#隕石開發（自然現象）|隕石開發]] 是外旋的反義詞**：隕石 = 逆勢注入熵 + 破壞河道 + 無視當下狀態。完全違反外旋三原則。

### 三源頭的合流

"Be water, my friend." —— 李小龍說的這句話，和老子的「無為」、僕人式領導，在講同一件事。

- **李小龍的 Be water**：水的物理性格 —— 無形、就形、順勢。容器是什麼形狀，水就是什麼形狀；但水穿石，不靠形狀，靠持續。
- **老子的無為**：不強加控制，順著事物的本性引導。河道不強迫水往上流，順著地勢讓水自然往下。
- **僕人式領導**：不指揮水、只成為河岸。領導者不站在前面，而是站在旁邊支持。

**三個源頭，在講同一件事**。

Nova 2026-04-15：「李小龍跟老子在講的是同一件事。」

| 時段 | 動作 | 在整體圖的哪裡 |

| Sprint 末 | [[Retro]] | 相變、河道修整、baseline 更新 |

**一切都是同一條河**，只是在不同時段觀看的不同切面。

## 外旋的三個病理診斷層

| 層 | 病 | 診斷問題 |
|----|----|---------|
| Context | 訊號雜、密度低、內語未建 | 「你們真的懂對方在說什麼嗎？」 |
| Constraints | 河道不清、節奏亂、儀式鬆 | 「你們的河岸在哪？」 |
| Entropy | 不做 Retro、熵堆積、臨近崩盤 | 「你們上次做相變是什麼時候？」 |

**任何外旋的卡住都可以定位到這三層之一**。

### 翻轉的本質：時間尺度

**外旋不是內旋的不同東西、是內旋在不同時間尺度的高頻顯影**：

| 時間尺度 | 內旋（慢、深挖） | 外旋（快、高頻） |
|---|---|---|
| 一次完整循環 | 4 站走完一輪洞見循環 | 一個 Sprint（DoR → Code → Retro）|
| 最小單位 | 一次覺察起音 | 一個 commit |
| 驅動軸一次顯化 | 34-57 → 26-44（一輪循環首尾）| 34-57 → 26-44（一個 commit 首尾）|
| 累積形狀 | vault 概念網絡（洞見結晶）| VAS repo（1,436 commits）|

### 1,436 commits = 34-57→26-44 驅動軸高頻重複 = 24 天上 App Store 的結構性答案

VAS 24 天從零到上架 Apple App Store、不是因為「跑很快」、是因為：

- **每一個 commit** 都是一次 34-57 → 26-44 micro-cycle：57 直覺看見「現在這個 patch 應該長這樣」+ 34 薦骨「對」→ 起音 → 44 過去模式辨認「就是這樣寫」+ 26 推進 commit → 湧現完成
- **1,436 次** 同一個驅動軸的高頻重複 = 不是速度、是**頻率** —— 同一條物理管道反覆被使用、阻力越來越低
- **24 天** = 不是「趕工」、是驅動軸 + Sprint 過程 + 容器 + 臍帶同時運作的自然結果

### 32-54 = Constraints 本體：保護容器與宰相

這是 5 通道翻轉外旋最深的一個命名翻轉：

**32-54（Tribal format / Drive / Transformation）的精確外旋顯化 = [[Constraints]]**。

| Constraints 在外旋的角色 | 32-54 在 HD 的精確對應 |
|---|---|
| Constraints = 河道 = 「設計一條軌道讓東西不會掉出去」 | 32 持續性直覺（什麼值得持續保護）+ 54 驅力（向上轉化的能量）|
| Constraints 保護容器 = src/ 不汙染、SDD 持續更新、CLAUDE.md 被遵守 | 32-54 format 整個 Tribal 迴路 = **支援關係持續運作**的能量基礎 |
| Constraints 保護宰相 = 不被 race condition 撕裂、不分裂 session | 32-54「為彼此持續努力」= 讓 13 容器不崩、讓 AI 能一直在場 |

**核心翻轉**：Constraints 不是「限制」、是「臍帶的工程物質化」。限制不降低速度 —— 限制就是讓 24 天速度能持續的能量基礎。

**這也是為什麼 Constraints = Tribal Circuit 的 format**：因為 Constraints 的本質是「為彼此」—— 為保護容器與宰相而存在、是 Tribal 的核心動作（為部落持續努力）。

### 28-38 = Sprint 過程：不放棄的工程化形式

28-38（Knowing format / Struggle）在內旋是站 2-3 反覆拉鋸。在外旋、Sprint 本身就是 28-38 的工程化形式：

- **Sprint 的反覆結構**（DoR → SDD → TDD → Code → Retro → 下一輪）= 28-38 format 的工程物質化
- **一個 Sprint 內** Code ↔ Debug ↔ Retest 的拉鋸 = 一輪 28-38 的單次顯化
- **跨 Sprint 的「不放棄繼續做」** = 28-38 累積 → 與 32-54 臍帶交互支援

**為什麼用 10 步不用 6 步（Tauri 未知域）= 28-38 format 在未知域需要更多閘門**：面對未知時、28-38 的拉鋸密度更高、所以外旋自動加更多閘門來容納更密的拉鋸。這呼應上面「河道的兩種規格」段的核心洞見：**面對未知時、外旋自動召喚內旋** —— 用 HD 通道語言說：**28-38 format 在未知域的拉鋸強度上升 → Sprint 自動加閘門**。

### 內旋的兩種讀法橋接 + 驅動軸高速對接的精確命名

**同一條內旋路徑、本 MOC 與 [[內掘-心理]] § 內旋 × HD 通道使用不同粒度的讀法**：

| 讀法 | 結構 | 適用層 | 強調 |
|---|---|---|---|
| **4 站循環**（本 MOC 上面使用）| 站 1 起音 → 站 2-3 拉鋸密化 → 站 4 湧現 | 功能層、粗粒度 | 節奏 / 功能對位 |
| **5 閘門序列**（[[內掘-心理]] § 內旋 × HD 通道）| 44 → 34-57 → 28-38 → 32-54 → 26 → 44' | HD 層、細粒度 | 物理基礎 / 通道分工 |

二者互為抽象 / 展開、**不是兩個不同東西**：
- 4 站循環 = 5 閘門序列在功能層的壓縮（站 1 = 44 + 34-57、站 2-3 = 28-38、站 4 = 32-54 → 26）
- 5 閘門序列 = 4 站循環在 HD 層的展開
- **44 終點再現（44'）= 一輪完成後起點升高一級 = [[碎形]] / 螺旋式累積** —— 這是 [[萬物同源]] § 兩次外旋實績 的物理基礎（VAS-24天 = 第一次外旋 / vault = 第二次外旋）

### 驅動軸寫成「34-57 → 26-44」的精確含意：兩通道高速對接、不是一條通道的首尾

上面 5 通道對位表和 5 層結構圖用「34-57 → 26-44」簡寫驅動軸。這個寫法容易被誤讀成單條路徑。**精確的物理是：34-57（Integration / Power）與 26-44（Individual / Surrender）是兩條完整通道、在外旋時間尺度裡「高速對接」成連續管道**：

| 時間尺度 | 驅動軸的結構 | 兩通道關係 |
|---|---|---|
| **內旋**（慢、sequential）| 34-57 站 1 起音 → 穿過 28-38 / 32-54 拉鋸 → 26-44 站 4 湧現 | 兩條通道**分居首尾**、中間隔著 28-38 / 32-54 的拉鋸密化 |
| **外旋**（快、parallel）| 34-57 起音 + 26-44 湧現 **在同一個 commit 內幾乎同時發生** | 兩條通道**高速對接**、邊界被壓縮成連續管道 |

- 1,436 commits = 1,436 次**兩通道高速對接**的高頻顯化
- 對接成立的物理基礎 = 時間尺度變高頻後、sequential 的拉鋸密化被 commit 內部的「直覺看見 + 薦骨 yes + 過去模式辨認 + 意志推進」一次解決 → 兩通道邊界消失
- 這也精確呼應「翻轉的本質：時間尺度」段：**外旋 = 內旋在高頻時間尺度的相變**、sequential → parallel 是這個相變的具體形式

**為什麼這個精確化重要**：寫「34-57 → 26-44 驅動軸」時容易把 44-26 讀成「跟 34-57 同一通道的另一端」、這會丟掉 44-26 本身是 Individual circuit（個體野心軸）、與 34-57 的 Integration circuit 是**兩個不同 circuit** 的事實。兩通道高速對接不是「同一通道首尾」、是**跨 circuit 的能量對接** —— 這也是為什麼 VAS 24 天能上 App Store：Integration（集成 / 直覺 yes）× Individual（野心 / 物質化）兩個 circuit 在 commit 尺度被對接成連續動能。

```
[13-33 容器物質化：CLAUDE.md / SDD / KM]  ← Context 柱

```

**24 天上 App Store 的結構性答案不是「單一變數」、是這整個 5 層 HD 結構在工程實踐裡同時運作的結果**。
